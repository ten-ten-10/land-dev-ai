# Land Scanner（土地開發 AI 補助決策系統）— Product Requirements Document

---

## 0. 品牌背景：TomoAid

**TomoAid** 是一個跨產業的 AI 解決方案平台品牌，核心理念是透過 AI 技術協助各領域專業人士（PM、HR、土地開發經理等）優化工作流程與決策效率。

### 品牌架構

| 子產品 | 垂直領域 | 核心功能 |
|---|---|---|
| **TomoHire** | HR 招募 | AI 輔助候選人篩選、媒合與面試流程優化 |
| **土地開發 AI** | 不動產開發 | 地號解析、容積試算、獎勵方案比較、ROI 初估 |

### TomoAid 三大核心目標

1. **跨產業 AI 賦能**：將 AI 應用延伸至土地開發、房地產分析等專業領域，解決資料龐雜與分析耗時的問題
2. **專業效率提升**：縮短從資料獲取到決策產出的週期，讓專業人士聚焦高價值判斷
3. **產品生態系**：建立統一品牌架構，針對不同產業需求發展垂直子產品

> **本文件範疇**：TomoAid 平台下的「土地開發 AI」子產品，聚焦台灣建設公司的土地評估決策場景。

## 1. 產品概述

### 1.1 背景

土地開發前期評估涉及大量法規查詢、容積試算、獎勵加乘計算與財務可行性分析，目前高度仰賴有經驗的開發人員人工判斷，效率低、門檻高、容易遺漏細節。本系統旨在透過 AI 輔助，將開發前期的複雜評估流程系統化、自動化，降低人工成本並提升決策品質。

台灣土地開發存在「先搶先贏」的競爭結構（買對地就贏一半），速度與準確性直接決定競爭優勢，本系統即為回應此核心痛點而設計。

### 1.2 產品目標

對齊 TomoAid 核心理念，本產品設定以下目標：

- 輸入土地基本資訊（地號、面積、使用分區），自動產出開發可行性初步評估報告
- 整合容積獎勵、都更、危老等補助機制，輔助決策最佳開發方案
- 縮短前期評估時間從**數天至數小時以內**（專業效率提升目標）
- 提供可追溯、可解釋的 AI 決策依據，降低人為疏漏風險

### 1.3 MVP 範疇

- **優先地區**：南台灣（高雄市優先，宜蘭縣、苗栗縣次之）
- **輸入**：土地地址 / 地號 → 自動查詢地號、使用分區、建蔽率/容積率（依各縣市都市計畫自治條例）
- **核心流程**：地籍定位 → 基準容積自動化 → 土地潛力評估（獎勵試算 + 套繪風險預警）→ ROI 初估

### 1.4 目標使用者

| 使用者角色 | 痛點 | 使用情境 |
|---|---|---|
| 土地開發評估人員（主要）| 法規繁雜、試算耗時 | 快速篩選與評估標的 |
| 建設公司專案經理 | 跨部門資訊彙整困難 | 主持評估會議、輸出報告 |
| 不動產投資分析師 | 財務試算模型建立費時 | ROI 試算與方案比較 |

## 2. 功能規格

### 2.1 四階段評估流程

```
篩選 Screening → 調查 Investigation → 評估 Evaluation → 簽約 Signing
```

| 階段 | 核心任務 | 自動化目標 |
|---|---|---|
| 篩選 | 過濾地點、面積、分區；自動排除畸零地、無建築線地號 | AI 自動篩選 + 紅旗標記 |
| 調查 | 謄本、他項權利、套繪管制、文化資產、地下障礙物 | 查詢自動化 + 風險摘要 |
| 評估 | 容積率、建蔽率、獎勵試算（危老/都更/容移）、量體模擬、土地殘值法 ROI | 全自動試算 + 方案比較 |
| 簽約 | 最高出價計算、談判支援 | 出價上限輸出 |

### 2.2 核心功能模組

#### F1 — 土地資訊輸入與解析
- 支援輸入地號（縣市、地段、地號）
- 自動抓取或手動輸入：土地面積、使用分區、建蔽率、容積率
- 支援上傳地籍圖、配置圖（PDF / JPG）進行 OCR 解析

**User Story**
> 身為土地評估人員，我希望輸入地址後系統能自動解析地號與分區資訊，這樣我不需要手動查詢多個政府系統。

**Acceptance Criteria (AC)**
- [ ] 輸入地址後，30 秒內回傳地號、使用分區、法定建蔽率/容積率
- [ ] 若查詢失敗，顯示明確錯誤原因與手動輸入入口
- [ ] OCR 解析地籍圖後，關鍵欄位（面積、地號）標記信心分數

#### F2 — 法規與容積試算
- 根據使用分區自動套用對應法規（建蔽率、容積率上限）
- 計算基準容積、獎勵容積（危老重建、都市更新、綠建築、智慧建築等）
- 核心公式：`總樓地板面積 = 基地面積 × 基準容積率 × (1 + 獎勵係數 + 移轉係數) + 免計容積`
- 輸出容積試算彙整表

**User Story**
> 身為評估人員，我希望系統自動計算不同獎勵方案下的可開發樓地板面積，這樣我能快速比較各方案的開發量體。

**Acceptance Criteria (AC)**
- [ ] 顯示基準容積 vs. 各獎勵方案容積的對照表
- [ ] 每項獎勵計算附上法規依據（條文名稱 + 出處）
- [ ] 試算結果誤差 < 5%（以人工驗算為基準）

#### F3 — 套繪風險預警
- 基於鄰地建案歷史與土地使用模式，預測套繪管制風險
- 以高／中／低風險分級呈現，標示高風險地號
- 提供風險原因說明、行動建議（如：建議申請地籍套繪圖）、預估處理時間與成本

**User Story**
> 身為土地評估人員，我希望系統在評估結果中自動標示套繪風險，這樣我能在談判前提早排除致命性法律問題。

**Acceptance Criteria (AC)**
- [ ] 高風險識別率 > 80%，誤報率 < 20%
- [ ] 風險說明以使用者可理解的語言呈現（附鄰地建案參考依據）
- [ ] 高風險案件提供具體行動建議，含預估時間與費用
- [ ] 明確標示此為「初步警示」，非正式法律意見

#### F4 — 補助方案比較
- 列出適用的補助與獎勵方案（危老、都更、容積移轉等）
- 各方案優劣比較（時程、成本、容積增益、風險）
- 推薦最適方案並附說明依據

**User Story**
> 身為建設公司 PM，我希望一眼看出各獎勵方案的利弊，這樣我能在評估會議中快速做出方案選擇。

**Acceptance Criteria (AC)**
- [ ] 顯示至少三種方案的並排比較表（含時程、成本、容積增益、主要風險）
- [ ] AI 推薦方案須附上 2–3 點理由說明
- [ ] 使用者可手動覆蓋推薦結果並備註原因

#### F5 — 財務可行性初估（Land Residual Value Method）
- 核心公式：`L = S − (C + M + P)`
  - L = 土地殘值（最高出價依據）
  - S = 總銷售收入
  - C = 建造成本
  - M = 管銷費用
  - P = 目標利潤
- 可開發樓地板面積推估
- 建造成本概估（單坪成本 × 面積）
- 銷售收入概估（市場行情 × 面積）
- 初步 ROI 與回收期試算

**User Story**
> 身為土地開發經理，我希望輸入市場行情與目標利潤後，系統計算出最高合理出價，這樣我在談判時有數字依據。

**Acceptance Criteria (AC)**
- [ ] 輸入欄位：單坪建造成本、目標容積、預估售價、目標利潤率
- [ ] 輸出：土地最高出價上限、ROI、回收期
- [ ] 支援敏感性分析（售價 ±10%、成本 ±10% 的影響）

#### F6 — 報告產出
- 自動生成前期評估報告（PDF / Markdown）
- 報告內容：土地基本資訊、法規摘要、容積試算、方案比較、財務初估、AI 建議
- 支援匯出與分享

**User Story**
> 身為 PM，我希望一鍵產出格式一致的評估報告，這樣我能直接提交給主管或合作方。

**Acceptance Criteria (AC)**
- [ ] 報告產出時間 < 3 分鐘
- [ ] 支援 PDF 匯出，格式含公司 Logo 欄位（TomoAid 品牌）
- [ ] 報告版本與產出時間戳記自動記錄

## 3. 使用者流程

```
輸入地號 / 上傳地籍圖
        ↓
系統解析土地資訊（分區、面積、法規）
        ↓
容積試算（基準 + 各項獎勵）
        ↓
補助方案比較與推薦
        ↓
財務可行性初估（Land Residual Value）
        ↓
產出評估報告
```

## 6. 非功能需求

| 項目 | 需求 |
|---|---|
| 效能 | 單筆評估報告產出 < 3 分鐘 |
| 準確性 | 容積試算誤差 < 5% |
| 安全性 | 土地資料不對外傳輸，本地或私有雲部署 |
| 擴充性 | 支援新增法規資料庫更新機制 |
| 可解釋性 | 每項 AI 判斷須附上依據說明（法規條文 / 計算邏輯） |

## 7. 技術架構（初步）

## 9. PM 工作規範

> 本節規範適用於 TomoAid 旗下所有產品（包含 TomoHire 與土地開發 AI），由負責多專案的 PM 制定，供 AI 助理（Claude）協作時遵循。

### 9.1 專案區分原則

- 每次對話請先辨識上下文屬於「**招募 / HR**」還是「**土地開發**」場景
- 避免混用兩個領域的術語或邏輯
- 若上下文不明，主動詢問確認後再作答

### 9.2 文件標準

所有 PRD 文件必須包含以下要素：

| 要素 | 說明 |
|---|---|
| **User Story** | 格式：身為 [角色]，我希望 [行為]，這樣我能 [目的] |
| **Acceptance Criteria (AC)** | 條列式，可測試、有明確通過條件 |
| **UI/UX 邏輯** | 說明使用者操作路徑、錯誤處理、邊界情境 |
| **Technical Feasibility** | 標記需與工程師確認的技術風險項目 |

### 9.3 溝通偏好

- **目標導向**：每次回應聚焦「解決什麼問題」，避免繞圈子
- **用語精簡**：善用表格與列點，避免冗長段落
- **術語保留**：保留專業術語的英文原稱，例如：
  - MVP、PRD、AC、ROI、FAR（Floor Area Ratio）
  - Land Analysis、Technical Feasibility、User Story
- **中文為主**：正文使用繁體中文，術語英文並陳

### 9.4 AI 協作規範

- Claude 在更新文件前，應先確認目前的文件版本與範疇
- 若使用者提供的資訊不完整，列點說明缺少哪些資訊，再請使用者補充
- 每次文件更新後，於文末更新「最後更新日期」與「文件版本號」
